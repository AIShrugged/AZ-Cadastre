# Лёгкий синхронный прототип вместо оркестрированной архитектуры

## Контекст

`docs/PRD.md` (исходный замысел) описывал тяжёлую production-архитектуру: Temporal для длительных workflow, WebSocket-gateway, PostgreSQL + S3/MinIO, Telegram-уведомления, 7-стадийный оркестратор и полный набор проверок A–D с ML-классификатором. Задача текущего этапа — **лёгчайший рабочий прототип без усложнений**, поэтому эта архитектура сознательно отвергнута в пользу тонкого вертикального среза.

## Решение

Собираем прототип в существующем pnpm-монорепо (`apps/web` — React 19 + Tailwind 4 + Redux Toolkit; `apps/core` — NestJS; `libs/contracts` — общие типы) со следующими сознательными упрощениями:

1. **Пайплайн — синхронный сервис в NestJS**, без Temporal, очередей и внешнего оркестратора. Стадии: разбивка PDF на страницы → классификация + извлечение полей (LLM) → Полнота (A) → Согласованность (C) → Отчёт.
2. **Распознавание — LLM-vision через OpenRouter** (мультимодальная модель), а не отдельный OCR-движок и не обучаемый классификатор. Страница уходит как изображение, модель возвращает тип и структурный JSON полей.
3. **Realtime-прогресс — SSE** (`@Sse()` + `EventSource`), а не WebSocket. Поток односторонний (сервер → инспектор), двунаправленный канал не нужен, socket.io не тянем.
4. **Хранение — SQLite + локальные файлы**, без PostgreSQL и S3/MinIO. Флаги и статусы инспектора переживают рестарт.
5. **Скоуп проверок — только A (полнота) и C (согласованность)**. B (корректность заполнения) и D (Dövriyyə/Ekspertiza: архивы MQS, пересечения соседей, ст. 8 закона) исключены — D требует внешних данных, которых нет.
6. **Роль инспектора — просмотр отчёта + отметка Флагов** («принято/отклонено», статус Дела). Без редактирования извлечённых полей и перезапуска проверок.
7. **Выход — веб-отчёт + выгрузка JSON**. Генерация Excel-чеклиста и PDF-отчёта отложена.
8. **Данные — синтетические**: одно «чистое» дело (Əliyeva, статус OK) и одно с подсаженными дефектами (Rusadze: фамилия Qusadze→Rusadze, расхождение площади, отсутствующий документ) для демонстрации A и C на контрасте.

## Последствия

- **Сегментация PDF наивная**: соседние страницы одного Типа склеиваются в один Документ. Точное определение границ документов внутри одного PDF — известный ML-риск и в прототипе не решается; на синтетических данных приемлемо.
- Порог расхождения площадей зашит как **±2%** (≤2% — норма, >2% — Флаг); текстовые поля сравниваются точно после нормализации.
- Качество извлечения зависит от чистоты синтетических документов — генерируем их достаточно «читаемыми», чтобы vision-модель работала стабильно.
- Возврат к оркестрированной архитектуре из PRD (Temporal/WS/Postgres/S3) остаётся возможным как отдельный этап; данный ADR его не закрывает, а откладывает.
